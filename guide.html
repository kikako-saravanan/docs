What is Module Federation?

Module Federation lets multiple independently deployed JavaScript applications share code at runtime, without npm publish, without build-time coupling. Think of it as microservices for the frontend.

Chapter 02 — Foundation

The Mental Model

Before touching any config, you need the right mental model. Get this right and everything else clicks.

Think in Containers

Every Module Federation app creates a container. A container exposes a get() and init() API. That's really all Module Federation is — a standardized way for containers to discover, initialize, and call each other.

CONCEPTcontainer interface
// Every remoteEntry.js or mf-manifest.json exposes:
interface Container {
  // Initialize this container with shared scope
  init(shareScope: SharedScope): Promise<void>;

  // Load an exposed module from this container
  get(module: string): Promise<() => any>;
}

The 4 Roles

ROLE 01
Host
The app loaded first. Initializes the shared scope. Consumes remote modules.
ROLE 02
Remote
An app that exposes modules. Must be running independently. Has its own build.
ROLE 03
Bidirectional
Can act as both host AND remote. Can consume while being consumed.
ROLE 04
Shared Scope
The global negotiation layer for dependencies. Prevents duplicate React instances.

What Happens at Runtime — Step by Step

STEP 1
Host boots
Webpack calls __webpack_require__.I, registering all shared deps.
→
STEP 2
Manifest fetched
Browser fetches mf-manifest.json (or remoteEntry.js) from remote URL.
→
STEP 3
init() called
Host passes its shareScope to the remote container so both sync shared deps.
→
STEP 4
get() called
The specific exposed module's factory is returned, then evaluated.

The Critical "Bootstrap" Insight

Module Federation requires that the host and remote negotiate shared deps asynchronously before any module can run. This is why you'll always see an index.js that just does import('./bootstrap'):

JSsrc/index.js
// ✅ CORRECT — async boundary lets MF negotiate shared deps first
import('./bootstrap');

// ❌ WRONG — synchronous import will cause "Shared module not available for eager consumption"
import React from 'react';
ReactDOM.render(...);
⚡ The Most Common Gotcha
Uncaught Error: Shared module is not available for eager consumption — This is caused by synchronously importing shared modules before the async negotiation completes. Always use the bootstrap pattern or set eager: true on the shared dep.
🧠 Quick Check: Why do we put our app code behind import('./bootstrap')?

Chapter 04 — Core Concepts

Host & Remote

These are the two fundamental roles in any MF architecture. Every app is one or both.

Host (Consumer)

The host is the app that consumes remote modules. It initializes the shared scope (the global pool of shared dependencies) and requests modules from remotes at runtime. The host boots first — its remoteEntry or mf-manifest.json is loaded and init() is called to register shared deps.

Remote (Provider)

The remote is the app that provides modules. It runs as its own web server, has its own build, and exposes specific modules via the exposes config. It does NOT need to know about the host at all — complete decoupling.

Bidirectional: Host + Remote at Once

A single app can be both. This is the most common real-world setup in large systems:

JSapp-B/webpack.config.js (host AND remote)
new ModuleFederationPlugin({
  name: 'appB',
  // Acting as REMOTE — exposing to others
  exposes: {
    './Checkout': './src/Checkout.tsx',
  },
  // Acting as HOST — consuming from others
  remotes: {
    designSystem: 'ds@http://localhost:3003/mf-manifest.json',
  },
  shared: { react: { singleton: true } },
})

Full Real-World Example: E-Commerce Shell

JSXshell/src/App.tsx
import React, { Suspense, lazy } from 'react';

// Dynamic import from 'product' remote → MF loads it lazily
const ProductList = lazy(() => import('product/ProductList'));
const Cart        = lazy(() => import('cart/Cart'));
const Header      = lazy(() => import('designSystem/Header'));

export default function App() {
  return (
    <Suspense fallback="Loading…">
      <Header title="My Store" />
      <ProductList category="electronics" />
      <Cart />
    </Suspense>
  );
}
💡 Pro Pattern: Error Boundary + Suspense
Always wrap remote imports in both <Suspense> and an <ErrorBoundary>. If the remote is down, you want a graceful fallback, not a crashed host app.
Chapter 05 — Core Concepts

exposes

The config that turns your app into a provider. You declare exactly what you want to share with the world.

Basic Syntax

JS
exposes: {
  // alias (what consumers will import) : actual file path
  './Button':        './src/components/Button.tsx',
  './Modal':         './src/components/Modal.tsx',
  './useAuth':       './src/hooks/useAuth.ts',
  './utils/format':  './src/utils/format.ts',
  '.':               './src/index.ts', // expose root
}

What Can You Expose?

Anything Webpack can bundle: React components, Vue components, utility functions, hooks, stores, full pages, even CSS modules. The exposed module becomes its own async chunk in the remote's build.

JSXsrc/components/Button.tsx (the exposed module)
// This file lives in the REMOTE app (product/:3001)
import React from 'react';

interface Props {
  label: string;
  onClick?: () => void;
  variant?: 'primary' | 'secondary';
}

export default function Button({ label, onClick, variant = 'primary' }: Props) {
  return <button className={`btn btn-${variant}`} onClick={onClick}>{label}</button>;
}
JSXshell/src/SomePage.tsx (the consumer)
// HOST app consuming the exposed Button
import React, { lazy, Suspense } from 'react';

// "product" is the remote name, "/Button" is the expose alias
const Button = lazy(() => import('product/Button'));

export default function Page() {
  return (
    <Suspense fallback="…">
      <Button label="Add to Cart" onClick={() => console.log('clicked')} />
    </Suspense>
  );
}

How exposes works under the hood

When you add an entry to exposes, Webpack's ContainerPlugin creates a new compilation entry for that module. It gets bundled into its own async chunk. The remoteEntry.js (or manifest) acts as the directory — it knows the chunk IDs of every exposed module and will load only the right chunk when the host calls container.get('./Button').

JSremoteEntry.js (auto-generated, simplified)
// Webpack auto-generates this. You never write it manually.
var product;
product = {
  get(module) {
    ensureChunk('src_components_Button'); // lazy loads the chunk
    return __webpack_require__('./src/components/Button.tsx');
  },
  init(shareScope) {
    // Syncs shared deps (react, etc.) with host
    __webpack_require__.S['default'] = shareScope;
  }
};
Chapter 06 — Core Concepts

remotes

How the host declares which external apps it wants to consume modules from.

Syntax

JS
remotes: {
  // localAlias: 'remoteName@URL_to_manifest_or_remoteEntry'

  // MF 2.0 style — points to mf-manifest.json
  product: 'product@http://localhost:3001/mf-manifest.json',

  // MF 1.0 style — still works in 2.0
  cart: 'cart@http://localhost:3002/remoteEntry.js',

  // Use localAlias that differs from remoteName
  ds: 'designSystem@http://localhost:3003/mf-manifest.json',
}

Dynamic Remotes at Runtime

Sometimes you don't know the URL at build time (e.g., it depends on env, feature flags, or A/B tests). Use the Runtime API to register remotes dynamically:

TSsrc/bootstrap.ts
import { registerRemotes } from '@module-federation/enhanced/runtime';

// Fetch config from your feature-flag service or API
const config = await fetch('/api/mf-config').then(r => r.json());

registerRemotes([
  { name: 'product', entry: config.productUrl },
  { name: 'cart',    entry: config.cartUrl },
]);

// Now you can use import('product/ProductList') as normal

Promise-based Remotes (MF 1.0 pattern)

JSwebpack.config.js
remotes: {
  // Promise-based: lets you async-resolve the URL before webpack uses it
  product: `promise new Promise(resolve => {
    const url = window.mfConfig?.productUrl || 'http://localhost:3001/remoteEntry.js';
    const script = document.createElement('script');
    script.src = url;
    script.onload = () => {
      resolve({
        get: (request) => window.product.get(request),
        init: (scope) => {
          try { window.product.init(scope); } catch(e) {}
        }
      });
    };
    document.head.appendChild(script);
  })`,
}
💡 MF 2.0 Preferred Approach
Instead of the verbose Promise syntax above, in MF 2.0 just use registerRemotes() from the Runtime API. It's cleaner and supports the Manifest protocol.
Chapter 07 — Core Concepts

shared

The most powerful and most misunderstood config. Get this wrong and you'll have multiple Reacts fighting each other.

What is shared?

When host and remote both depend on React, without shared you'd download React twice. With shared, they negotiate at runtime: "I have React 18.2 — do you want to use mine?" If versions are compatible, only one copy runs.

SHARED SCOPE NEGOTIATION
HOST
react@18.2.0
react-dom@18.2.0
→ shareScope →
SHARED SCOPE
{ react: { "18.2.0": factory } }
← syncs ←
REMOTE
react@18.2.0
(already loaded? skip!)

All shared Options

singleton
boolean
Only one instance ever. If versions conflict, warn but use the highest loaded version. Essential for React, React-DOM, stores.
requiredVersion
string (semver)
e.g. "^18.0.0". Specifies what version range is acceptable. If mismatch, MF will warn and may fallback to bundling its own.
eager
boolean
Don't put in async chunk. Include synchronously in initial bundle. Use when you MUST have a dep available before the async boundary.
strictVersion
boolean
Throw an error (not just a warning) if the version doesn't match. Safer but less flexible.
shareScope
string
Default is "default". You can create isolated scopes to avoid cross-contamination between unrelated MF groups.
version
string
Override the version reported. Useful in monorepos where package.json version isn't the truth.

Comprehensive shared Config Example

JSwebpack.config.js
shared: {
  // Bare minimum — just declare it
  lodash: 'lodash',

  // With options
  react: {
    singleton: true,
    requiredVersion: '^18.0.0',
    // eager: true  ← use only in root host to avoid "Shared module not available" error
  },
  'react-dom': {
    singleton: true,
    requiredVersion: '^18.0.0',
  },

  // Your own shared library
  '@myorg/design-tokens': {
    singleton: true,
    requiredVersion: '>=1.0.0',
  },

  // State management — MUST be singleton or stores break
  '@reduxjs/toolkit': {
    singleton: true,
    requiredVersion: '^2.0.0',
  },
  'react-redux': {
    singleton: true,
    requiredVersion: '^9.0.0',
  },
}

The Version Conflict Matrix

ScenariosingletonstrictVersionResult
Host: React 18 / Remote: React 18truefalse✅ Shared perfectly
Host: React 18 / Remote: React 17truefalse⚠️ Warning, uses React 18
Host: React 18 / Remote: React 17truetrue❌ Error thrown

⚡ The Two-React Problem
If React is loaded twice, you'll see "Invalid hook call" or "React is not defined" — cryptic errors that are hard to debug. The fix: ensure both host and remote declare react: { singleton: true } and have compatible versions.

Chapter 08 — MF 2.0 Features

mf-manifest.json

The game-changer in MF 2.0. A self-describing contract file that enables types, preloading, versioned deployments, and more.

🆕 New in MF 2.0
The manifest file is automatically generated by @module-federation/enhanced. You get it for free — just set your remote URL to point at mf-manifest.json.

What's Inside mf-manifest.json

JSONdist/mf-manifest.json (auto-generated)
{
  "id": "product",
  "name": "product",
  "metaData": {
    "buildInfo": { "buildVersion": "1.0.0", "buildName": "product" },
    "remoteEntry": {
      "name": "remoteEntry.js",
      "path": "",
      "type": "module"
    },
    "types": {
      "path": "",
      "name": "@mf-types.zip", // ← TypeScript types ZIP!
      "api": "@mf-types-api.json"
    },
    "publicPath": "http://localhost:3001/"
  },
  "shared": [
    { "id": "product:react", "name": "react", "version": "18.2.0" }
  ],
  "remotes": [],
  "exposes": [
    {
      "id": "product:./Button",
      "name": "./Button",
      "assets": {
        "js": { "async": ["src_Button_tsx.js"], "sync": [] },
        "css": { "async": [], "sync": [] }
      },
      "path": "./Button"
    }
  ]
}

What the Manifest Enables

📦
Type Sharing
The types.name field points to a @mf-types.zip — the consumer can automatically download TypeScript types from the remote.
⚡
Preloading
The assets field per expose lists all chunks. The Runtime can prefetch them before you even call import().
🚀
Versioned Deployment
Platforms like Zephyr Cloud use manifest metadata for canary releases, A/B testing, and rollback — without redeploying the host.
🔍
Dependency Analysis
The Chrome DevTool reads the manifest to visualize which shared deps are used, from which version, and where collisions exist.

Enabling Manifest in Config

JSwebpack.config.js
const { ModuleFederationPlugin } = require('@module-federation/enhanced/webpack');

module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: 'product',
      exposes: { './Button': './src/Button.tsx' },
      shared: { react: { singleton: true } },
      // Manifest is auto-generated! Optional custom config:
      manifest: {
        fileName: 'mf-manifest.json',  // default
        filePath: '',                   // relative to output.path
        disableAssetsAnalyze: false,   // set true to speed up complex builds
      },
    }),
  ],
};
Chapter 09 — MF 2.0 Features

Federation Runtime

The standalone JavaScript runtime that powers MF 2.0. Decoupled from any build tool — works with Webpack, Rspack, Vite, or plain script tags.

Why a Standalone Runtime?

In MF 1.0, the federation logic was baked into Webpack's runtime. You couldn't use it without Webpack, and you couldn't extend it. MF 2.0 extracts this into @module-federation/runtime — a separate, embeddable, pluggable runtime.

BASH
npm install @module-federation/enhanced
# Runtime is included in @module-federation/enhanced/runtime

Standalone Runtime (no build tool needed)

TSruntime-only usage
import { init, loadRemote, loadShare } from '@module-federation/enhanced/runtime';

// 1. Initialize the runtime
init({
  name: 'shell',
  remotes: [
    {
      name: 'product',
      alias: 'product',
      entry: 'http://localhost:3001/mf-manifest.json',
    },
    {
      name: 'cart',
      entry: 'http://localhost:3002/remoteEntry.js', // MF 1.0 compat
    },
  ],
  shared: {
    react: {
      version: '18.2.0',
      scope: 'default',
      lib: () => require('react'), // factory function
      shareConfig: { singleton: true, requiredVersion: '^18.0.0' },
    },
  },
});

// 2. Load a remote module
const { default: Button } = await loadRemote<{ default: React.FC }>('product/Button');

// 3. Load a shared module
const reactFactory = await loadShare('react');
const React = reactFactory();

createInstance — Multiple Federation Instances

TS
import { createInstance } from '@module-federation/enhanced/runtime';

// When you need isolated MF instances (e.g., micro-frontend within micro-frontend)
const mf = createInstance({
  name: 'isolatedHost',
  remotes: [{ name: 'sub1', entry: 'http://localhost:8080/mf-manifest.json' }],
});

const mod = await mf.loadRemote('sub1/util');
mod.add(1, 2, 3);

// Debug: inspect all instances in browser console

preloadRemote — Solve the Waterfall

The classic problem: Shell boots → fetches manifest → fetches remote chunks → finally renders. That's 3 serial round trips. Preloading collapses them:

TSsrc/bootstrap.ts
import { init, preloadRemote } from '@module-federation/enhanced/runtime';

init({ name: 'shell', remotes: [{ name: 'product', entry: 'http://localhost:3001/mf-manifest.json' }] });

// Kick off preloading BEFORE user navigates to that page
preloadRemote([
  {
    nameOrAlias: 'product',
    exposes: ['./ProductList', './ProductDetail'],
    resourceCategory: 'all',       // 'all' | 'sync'
    prefetchInterface: true,        // also prefetch the manifest
  }
]);

// By the time user clicks "Products", the chunks are already in cache!

loadShare with Custom Resolver

TS
import { loadShare } from '@module-federation/enhanced/runtime';

// Advanced: pick a specific version from shared scope
const reactFactory = await loadShare('react', {
  resolver: (sharedOptions) => {
    // sharedOptions = all registered versions of react
    return sharedOptions.find(i => i.version === '18.2.0') ?? sharedOptions[0];
  }
});
const React = reactFactory();

Runtime Hooks & Plugin System

The most powerful extensibility in MF 2.0. Hook into any part of the loading lifecycle.

Available Hooks

Plugin Structure

TSplugins/auth-plugin.ts
import type { FederationRuntimePlugin } from '@module-federation/enhanced/runtime';

export default function authPlugin(): FederationRuntimePlugin {
  return {
    name: 'auth-plugin',

    // Add auth token to every manifest fetch
    fetch(url, init) {
      const token = localStorage.getItem('jwt');
      return fetch(url, {
        ...init,
        headers: { ...init?.headers, Authorization: `Bearer ${token}` },
      });
    },
  };
}

Fallback Plugin — Production Essential

TSplugins/fallback-plugin.ts
import type { FederationRuntimePlugin } from '@module-federation/enhanced/runtime';
import FallbackButton from './FallbackButton';

export default function fallbackPlugin(): FederationRuntimePlugin {
  return {
    name: 'fallback-plugin',

    errorLoadRemote({ lifecycle, id, error }) {
      console.warn(`[MF] Failed to load ${id} at ${lifecycle}:`, error);

      switch (lifecycle) {
        case 'afterResolve': // Manifest fetch failed
          return {
            id,
            name: id,
            metaData: { remoteEntry: { name: 'fallback', path: '', type: 'global' } },
            shared: [], remotes: [], exposes: [],
          };

        case 'onLoad': // Module chunk failed to load
          return {
            factory: () => ({ default: FallbackButton })
          };

        default:
          return void 0;
      }
    }
  };
}

Registering Plugins

TSsrc/bootstrap.ts
import { init } from '@module-federation/enhanced/runtime';
import authPlugin from './plugins/auth-plugin';
import fallbackPlugin from './plugins/fallback-plugin';

init({
  name: 'shell',
  remotes: [{ name: 'product', entry: 'http://localhost:3001/mf-manifest.json' }],
  plugins: [authPlugin(), fallbackPlugin()],
});
Chapter 12 — MF 2.0 Features

Dynamic Type Hints

One of MF 2.0's killer features. Auto-generate and auto-sync TypeScript types from remotes — no manual type sharing.

The Problem it Solves

In MF 1.0, if Remote A exposes a Button component and Host B uses it, Host B has no TypeScript types for Button unless Remote A publishes an npm package or you manually copy types. With large teams this becomes a nightmare.

How it Works in MF 2.0

BUILD TIME (Remote)
Types Generated
MF plugin runs tsc on exposed modules, bundles types into @mf-types.zip
→
MANIFEST
Types Referenced
mf-manifest.json has metaData.types.name pointing to the zip file
→
HOST DEV SERVER
Types Downloaded
MF plugin fetches the zip, extracts to @mf-types/ in host's root
→
IDE
TypeScript ❤️
Your editor now has full types for import('product/Button')

Configuration

JSwebpack.config.js (Remote — generates types)
new ModuleFederationPlugin({
  name: 'product',
  exposes: { './Button': './src/Button.tsx' },
  shared: { react: { singleton: true } },
  // dts: true is the default in @module-federation/enhanced
  // Explicit config:
  dts: {
    generateTypes: true,        // generate @mf-types.zip
    compileInChildProcess: true, // don't block the main build
    tsConfigPath: './tsconfig.json',
  },
})
JSwebpack.config.js (Host — consumes types)
new ModuleFederationPlugin({
  name: 'shell',
  remotes: { product: 'product@http://localhost:3001/mf-manifest.json' },
  dts: {
    consumeTypes: true,         // download & extract types on dev server start
  },
})
JSONtsconfig.json (Host — tell TS where the types are)
{
  "compilerOptions": {
    "paths": {
      "product/*": ["./@mf-types/product/*"]
    }
  }
}

⚡ publicPath: 'auto' is Critical
Always set output.publicPath: 'auto' in remote apps. Without it, the remote's chunks will be fetched from the host's URL, causing 404 errors.
Chapter 14 — Advanced

Performance

MF introduces new performance challenges. Here's how to master them.

The Waterfall Problem

Without preloading, MF has a sequential network request chain:

TIMELINE
0ms   → GET /mf-manifest.json (shell's own)
100ms → GET http://remote1:3001/mf-manifest.json
200ms → GET http://remote1:3001/remoteEntry.js
300ms → GET http://remote1:3001/src_Button_tsx.js
400ms → 💥 React finally renders the remote component

With preloadRemote() these happen in parallel:
0ms   → GET manifest + GET remoteEntry + GET chunks  (all at once)
100ms → 💥 React renders immediately

Shared Tree Shaking

MF 2.0 supports tree-shaking across shared modules. If the remote only uses 20% of a shared utility library, only those exports get bundled:

JSwebpack.config.js
shared: {
  '@myorg/utils': {
    singleton: true,
    // requiredVersion + import path = tree shaking eligible
    import: '@myorg/utils',
    requiredVersion: '^1.0.0',
  }
}

Pattern 3: Cross-MFE Communication

Micro-frontends should NOT directly call each other. Use a shared event bus or shared store exposed from a common remote.

TSshared-state/src/eventBus.ts
// Exposed from a 'sharedState' remote
type Handler = (data: unknown) => void;
const listeners = new Map<string, Set<Handler>>();

export const emit = (event: string, data: unknown) => {
  listeners.get(event)?.forEach(h => h(data));
};

export const on = (event: string, handler: Handler) => {
  if (!listeners.has(event)) listeners.set(event, new Set());
  listeners.get(event)!.add(handler);
  return () => listeners.get(event)!.delete(handler); // unsubscribe
};

Pattern 4: Canary Releases

Deploy two versions of a remote, route a % of users to the new version — without touching the host.

TSshell/src/bootstrap.ts
import { init, registerRemotes } from '@module-federation/enhanced/runtime';

const isCanaryUser = Math.random() < 0.1; // 10% canary

init({ name: 'shell', remotes: [] });

registerRemotes([{
  name: 'product',
  entry: isCanaryUser
    ? 'http://cdn.myco.com/product/v2/mf-manifest.json'
    : 'http://cdn.myco.com/product/v1/mf-manifest.json',
}]);

Troubleshooting

Every error you'll encounter in production, and exactly how to fix it.

Runtime Errors

ErrorRoot CauseFix
Shared module is not available for eager consumption Synchronous import before async negotiation Add import('./bootstrap') in index.js
Invalid hook call Two copies of React loaded react: { singleton: true } on all apps
Cannot find module 'remoteName/expose' Remote not declared in remotes Add to remotes config and rebuild
404 on remoteEntry.js Wrong URL, remote not running, wrong publicPath Verify URL, check remote dev server, set publicPath: 'auto'
Stale module after hot reload Module cache not invalidated Add output.uniqueName in remote config
CORS error on manifest fetch Remote server doesn't allow cross-origin Add CORS headers to remote dev server / CDN

Build Errors

ErrorRoot CauseFix
Circular dependency App A remotes App B, B remotes A MF supports circular deps — ensure init order is correct
Type errors on remote imports Types not downloaded or tsconfig paths missing Enable dts.consumeTypes: true, add tsconfig paths
Build succeeds but chunks 404 publicPath is wrong Set output.publicPath: 'auto' in remote
